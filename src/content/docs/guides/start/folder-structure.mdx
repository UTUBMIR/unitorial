## 🏗️ Why does folder structure matter?

#### Proper folder organization:
* Reduces time spent searching for assets.
* Improves team collaboration.
* Makes projects easier to scale.



## 🎯 Core idea
Many beginners create separate folders and subfolders by asset type (`Scripts`, `Prefabs`, `Materials`). This often creates an *"**illusion**"* of order.
While it may seem logical, these elements usually work together, so splitting strictly by type is rarely optimal.

This can work at first, but over time it introduces more chaos as unrelated things pile up in the same type folders.
For example, a `Scripts` folder may contain player scripts, enemy scripts, and UI scripts.
These are not tightly related to each other, but they still end up in one place, which hurts organization.

<details open>
    <summary>Example of a poor folder structure</summary>

    <FileTree>
    - Scripts
        - Controllers
            - PlayerController.cs
            - EnemyAI.cs
        - Factories
            - PlayerFactory.cs
            - EnemyFactory.cs
        - UI
            - Entities
                - PlayerUI.cs
                - EnemyUI.cs
            - Button.cs
    - Prefabs
        - Player.prefab
        - Enemy.prefab
        - Button.prefab
    - Sounds
        - EnemySound.mp3
        - AttackSound.mp3
    </FileTree>
</details>

:::note
There are many valid ways to structure folders, and there is no single correct solution.

The approach below is one practical pattern that works well for many projects. Adapt it to your needs.
:::

## 📂 Feature-based approach
In a feature-based approach, folders are organized by game functionality *"**features**"*, not by asset type.

Instead of separate top-level folders like `Scripts`, `Prefabs`, and `Sounds`, group things by features such as `Player`, `Enemy`, and `UI`.

Each feature folder contains all assets related to that feature, regardless of type.
For example, a `Player` folder may include controller scripts, prefabs, and player-related sounds.

This keeps everything related to one feature together, making maintenance and navigation much easier.

<details open>
    <summary>Example of a good folder structure</summary>

    <FileTree>
    - Features
        - Player
            - PlayerController.cs
            - PlayerFactory.cs
            - PlayerUI.cs
            - Player.prefab
            - PlayerSound.mp3
        - Enemy
            - EnemyAI.cs
            - EnemyFactory.cs
            - EnemyUI.cs
            - Enemy.prefab
            - EnemySound.mp3
    - UI
        - Button.cs
        - Button.prefab
    </FileTree>
</details>

:::note
A feature-based structure solves the resource *"ownership"* problem.
But folder hierarchy still matters.
:::

:::tip
If needed, use type-based subfolders inside *"feature"* folders for extra clarity.

Like this:
<FileTree>
- Player
    - ... **General** player-related assets
    - Prefabs
        - ... **Player prefabs**
    - Sounds
        - ... **Player sounds** or action sounds
</FileTree>
:::

## 📌 Summary

- Organize folders by **purpose/feature**, not only by asset type.
- Use **type-based subfolders** inside a feature when additional structure is helpful.
