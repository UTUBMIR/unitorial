## 🧭 Навіщо продумувати систему меню?

#### Якісна система меню:
* Створює хороше перше враження про гру.
* Полегшує навігацію для гравця.
* Дозволяє повторно використовувати UI-екрани та компоненти.
* Спрощує підтримку `Main Menu`, `Pause Menu`, `Settings` та діалогових вікон.

## 🎯 Головна ідея
Поширена помилка - створювати кожен екран меню як окремий ізольований набір кнопок і скриптів, які напряму відкривають один одного.

Спочатку це виглядає швидко та зручно, але з часом така структура починає ламатися:
кнопки містять логіку переходів, налаштування зберігаються де завгодно, а `Main Menu`, `Pause Menu` і `Settings` поводяться як три різні системи.

Кращий підхід - думати про меню як про **єдину систему навігації**, де екрани, модальні вікна, введення гравця та переходи працюють разом.

## 🧱 Загальні підходи

### 1. Заміна одного екрана іншим
Найпростіший підхід: у вас є кореневий контейнер, у якому в один момент часу активний лише один екран.

Добре підходить для:
* `Main Menu`
* `Settings`
* простих навчальних або невеликих проєктів

Мінуси:
* складно працювати з модальними вікнами
* незручно повертатися назад
* важче повторно використовувати один і той самий екран у різних місцях

### 2. Стековий підхід
Екрани та вікна відкриваються поверх один одного за принципом стеку: `Push` для відкриття, `Pop` для повернення назад.

Добре підходить для:
* `Pause Menu`
* `Settings` поверх іншого меню
* вікон підтвердження, наприклад `Are you sure?`

Плюси:
* природна кнопка `Back`
* легко відкривати модальні вікна поверх поточного екрана
* зручно ділити меню на шари

### 3. Підхід через стани або flow
У складніших проєктах меню краще розглядати як набір станів або сценаріїв переходу: `Boot -> Main Menu -> Save Select -> Settings -> Gameplay -> Pause`.

Добре підходить для:
* великих ігор
* меню з багатьма умовами
* проєктів, де важлива передбачуваність переходів

Плюси:
* менше випадкових переходів
* легше тестувати
* простіше контролювати, що саме можна відкрити в конкретний момент

:::tip
На практиці ці підходи часто комбінують:

`Main Menu` працює як набір екранів, а поверх нього вже використовують стек для модальних вікон і підменю.
:::

## 🧩 Що робить систему меню справді сильною?

### Єдина точка керування
Бажано мати один `MenuController` або `MenuManager`, який:
* відкриває екрани
* закриває їх
* керує `Back`
* запускає анімації переходів
* перемикає ввод між мишею, клавіатурою та геймпадом

### Розділення відповідальностей
Не змішуйте все в одному класі.

Корисно розділяти:
* **View**: кнопки, анімації, візуал
* **Logic**: відкриття, закриття, навігація
* **Data/Config**: список пунктів меню, шляхи переходів, дефолтний фокус, локалізація

### Повторне використання екранів
`Settings`, `Confirmation Dialog`, `Keybind Screen` або `Save Slot Selector` часто потрібні в різних місцях.

Якщо ці екрани не прив'язані жорстко до одного меню, ви зможете використовувати їх і з `Main Menu`, і з `Pause Menu`.

### Передбачувана навігація
Гравець не повинен гадати:
* що буде після натискання `Escape`
* куди поверне кнопка `Back`
* який елемент зараз у фокусі

Особливо важливо одразу продумати навігацію для клавіатури та геймпада, а не лише для миші.

У Unity для цього вже є готова база:
* `EventSystem` відповідає за поточний вибраний UI-елемент
* у `Selectable`-елементів (`Button`, `Slider`, `Toggle`) є вбудована `Navigation`
* `Input System UI Input Module` або `Standalone Input Module` допомагають підключити навігацію з клавіатури, геймпада та миші до однієї системи

Тобто вам не обов'язково вигадувати навігацію з нуля, але важливо **свідомо налаштувати** фокус, напрямки переходу та поведінку кнопки `Back`.

### Однакові правила переходів
Анімації, затримки, звуки підтвердження та логіка відкриття меню мають бути послідовними.

Коли кожен екран реалізований по-своєму, меню швидко починає відчуватися "дешево", навіть якщо візуально воно красиве.

## 🎮 Приклади з Unity

### Main Menu
У Unity `Main Menu` часто роблять як окремий `Canvas` або набір панелей усередині одного `Canvas`.

Для невеликого проєкту цього достатньо: `Play`, `Settings`, `Quit`, а перемикання між підекранами можна робити через активацію/деактивацію панелей.

### Settings
Екран налаштувань зручно будувати з тих самих стандартних UI-компонентів Unity:
* `Slider` для гучності
* `Toggle` для опцій типу `Fullscreen`
* `Dropdown` або `TMP_Dropdown` для вибору роздільної здатності чи якості

Тут важливо не лише показувати зміну значень, а й одразу вирішити, які налаштування застосовуються миттєво, а які лише після `Apply`.

### Pause Menu
У Unity `Pause Menu` часто є просто ще одним шаром меню поверх гри.

Тут добре працює стековий підхід: відкрили `Pause Menu`, поверх нього відкрили `Settings`, натиснули `Back` і повернулися на попередній екран, не втрачаючи контекст.

### Модальні вікна
Для `Confirmation Dialog` на кшталт `Exit to Main Menu?` або `Delete Save?` Unity-підхід теж зручний:
окрема панель, яка блокує взаємодію з нижнім меню, поки гравець не підтвердить або не скасує дію.

У таких випадках корисно одразу визначати дефолтний вибраний елемент, щоб після відкриття вікна фокус уже був на `Cancel` або `Confirm`.

## 📌 Підсумок

- Думайте про меню як про **систему навігації**, а не просто набір кнопок.
- Для простих випадків підходить **заміна одного екрана**, для гнучкості краще мати **стек** або **стани**.
- Найкращий результат дає **єдиний контролер**, повторно використовувані екрани та передбачувані правила навігації.
