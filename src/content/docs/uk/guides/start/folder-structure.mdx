## 🏗️ Навіщо потрібна структура папок?

#### Правильна організація папок:
* Зменшує час на пошук потрібних ресурсів.
* Полегшує співпрацю в команді.
* Зпрощує масштабування проєкту.



## 🎯 Головна ідея
Багато новачків створюють окремі папки і підпапки для кожного типу ресурсу (`Scripts`, `Prefabs`, `Materials`), цей підхід часто створює *"**іллюзію порядку**"*.
Хоча він і здається логічним - насправді всі ці елементи працюють разом, а тому немає сенсу розділяти їх за типом.

Спочатку такий підхід може працювати, але з часом він стає приносити все більше і більше хаосу, оскільки все змішується в одній папці.
Наприклад, у папці `Scripts` можуть бути скрипти для гравця, ворогів та UI.
Всі ці елементи не зв'язані між собою, але вони всі знаходяться в одній папці, що взагалі не допомагає в організації.

Спосіб що буде показано далі - це лише один з можливих підходів, який добре працює для багатьох проєктів, але ви завжди можете адаптувати його під свої потреби.
:::


## 📂 Фічевий підхід
У фічевому підході папки організовані за функціоналом або *"**фічами**"* гри, а не за типом ресурсу.

Наприклад, замість того, щоб мати окремі папки для `Scripts`, `Prefabs` та `Sounds`, ми можемо організувати папки за фічами, такими як `Player`, `Enemy` та `UI`.

Кожна з цих папок буде містити всі ресурси, пов'язані з цією фічею, незалежно від їх типу.
Наприклад, папка `Player` може містити скрипти для контролера гравця, префаби гравця та звуки, пов'язані з гравцем.

Цей підхід допомагає зберігати всі ресурси, пов'язані з певною фічею, разом, що полегшує їх пошук та підтримку.
<details open>
    <summary>Приклад хорошої структури папок</summary>

    <FileTree>
    - Features
        - Player
            - PlayerController.cs
            - PlayerFactory.cs
            - PlayerUI.cs
            - Player.prefab
            - PlayerSound.mp3
        - Enemy
            - EnemyAI.cs
            - EnemyFactory.cs
            - EnemyUI.cs
            - Enemy.prefab
            - EnemySound.mp3
    - UI
        - Button.cs
        - Button.prefab
    </FileTree>
</details>

:::note
Фічевий підхід вирішує проблему *"призначення"* ресурсів.
Але ієрархія папок всеодно грає важливу роль.
:::

:::tip
При необхідності - в папках для *"фіч"* ми можемо мати підпапки для типів, щоб ще більше організувати ресурси.

Отак:
<FileTree>
- Player
    - ... **Загальні** речі пов'язані з гравцем
    - Prefabs
        - ... **Префаби** граців
    - Sounds 
        - ... **Звуки** пов'язані з гравцем або його діями
</FileTree>
:::

## 📌 Підсумок

- Варто організовувати папки за **призначенням**, а не за типом ресурсів.
- Бажано **використовувати підпапки з типами** для ще більшої структури.
